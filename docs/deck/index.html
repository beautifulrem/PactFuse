问题

AI Agent 正在自己花钱

它能下单、签名、付款。可是它被骗了怎么办?它要买的东西在付款前一秒变了怎么办?谁来拦?

被注入

提示注入让 Agent 把钱付给错误的合约地址,它自己不会怀疑。

来源失效

报价时来源是安全的,付款前却被吊销/挑战,钱照样打出去。

付款拿不到货

钱已上链,交付却没发生,链上不可逆。

80% 的组织踩过 agent 的坑(McKinsey ↗)。链上交易没有"撤销"按钮,发现得再快也没用。

真实会发生的事

你的 Agent买了一份过期数据

你的交易 Agent 要从链上数据商买一份 24 小时实时行情喂价。

  • 调查时:Agent 检查了供应商,Base 上有注册,数据质量评分正常,报价 500 mUSD / 24h。决定买入。
  • 30 分钟后:攻击者劫持了供应商的 API 端点,把真实行情替换成了操纵过的价格数据。链上监控者把该数据源标记为Challenged。
  • 付款时:Agent 还在用 30 分钟前的检查结果。它付了 500 mUSD,拿到的是假行情,接下来所有交易决策都建立在错误数据上。

88% 的组织踩过这种坑。beam.ai 2026 ↗

现有方案的盲区

决策时 ≠ 付款时

  • 白名单 / LLM / 风控:管"能不能发起"这个操作
  • Web3 交易没法撤,中间那几分钟来源可能变了
  • PactFuse:管"钱真的打出去那一刻"

只靠 CAW 的链下判断和 LLM 的软限制,无法解决链上数据源的问题。

为什么光靠 CAW + LLM 不行

它们管得了决策,管不了付款那一秒

我们在哪

PactFuse 的定位

  • 左上角:付款时介入 + 链上判断。只有 PactFuse 在这个位置。
  • LLM 风控 / 打分:决策时判断,链下执行。判断能力受限于 LLM,可被投毒。
  • 审计工具:决策时检查,但拦不住交易本身。
  • 多签/MPC:链上能拦,但只管"谁签了",不管"来源还安不安全"。
解法

付款前,在链上熔断

别的项目往往只触及链下决策,却忘了决策时和结算时是两个时间点。

熔断

来源不安全 → 闸门跳开,0 转移

结算

来源干净 → 授权、支付、交付

拒绝

目标不在白名单 → 链下拒绝

关键代码

"能不能付"写在合约里

① 闸门:来源不 Active 就 revert,支付无法落地

// ProcurementGate (示意)
function settleSpend(bytes32 src, ...) {
  require(
    sourceState[src] == Active,
    "source not active"   // 跳闸 → 0 转移
  );
  /* …授权 + 支付 + 交付… */
}

② 浏览器 keyless 复核:eth_call 直读链上状态

// chain.mjs (示意)
const data = "0x447c24c0" + pad(srcHash); // sourceState(bytes32)
const r = await rpc("eth_call", [{
  to: registry, data
}, "latest"]);
// 0=Unknown 1=Active 2=Challenged 3=Revoked

内置传输中断模拟(?fail=1):即使执行中断,资金仍留在原地,重试即可恢复。
