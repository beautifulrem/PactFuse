问题

AI Agent 正在自己花钱

它能自己下单、签名、付款。可一旦它被骗、或它买的东西在付款瞬间已经变质,谁来拦?

被注入

提示注入让 Agent 把钱付给错误的合约地址,它自己不会怀疑。

来源失效

报价时来源是安全的,付款前却被吊销/挑战,钱照样打出去。

付款拿不到货

钱已上链,交付却没发生,链上不可逆。

80% 的组织已踩过 AI agent 风险行为;出问题的瞬间钱已经在路上,链上不可逆,事后追不回。来源:McKinsey · OWASP Agentic Top 10

为什么现有做法不够

检测 ≠ 强制

  • 风控/打分/告警:能发现风险,但只是"提醒",拦不住已经签出去的交易。
  • 事后审计:钱上链后才发现,链上不可逆。
  • 搜索/检索结果不是结算级真相,Agent 不该拿它直接付款。

行业正转向确定性强制(deterministic enforcement):OWASP Agentic Top 10、Visa/Mastercard 都在补"验证层"。PactFuse 把这道强制直接放到链上。

解法

付款前,在链上熔断

PactFuse 是一个 fail-closed 的链上断路器:在支付落地的那一刻重新核验来源,不安全就切断回路,一个 atomic 都不动。

熔断

来源被挑战 → 闸门跳开,所有钉住它的支付在付款前被切断。

结算

来源干净 → 在策略边界内授权、支付、交付 artifact。

拒绝

目标不在 Pact 白名单 → 任何东西上链前直接拒绝。

默认即安全(fail-closed):有疑问就不放行,而不是先付了再说。

怎么做 · 架构

一次支付的三道关卡

付款前的决策流

每一笔支付,先过这道判断

关键在 读 sourceState 这一步:它是链上确定性判断,不是模型猜测。不安全就 revert,钱落不了地。

三个场景 · 一键复现

同一条支付线,三种结局

TRIP · 熔断

不安全来源 → 自动熔断

挑战›链上确认›闸门跳闸›0 转移

来源在报价后被挑战,断路器跳开,钉住它的支付在付款前被切断。

已保护 · 0 资金转移

SETTLE · 结算

新鲜来源 → 结算并交付

授权›额度↑›支付›交付 artifact

来源干净,闸门在策略内授权、支付、释放 artifact。

DENY · 拒绝

错误目标 → 策略拒绝

错误目标›策略不符›链下拒绝›0 上链

目标不在 Pact 白名单,CAW 在任何东西上链前拒绝。

现实场景

把三场景映射到真实采购

数据/API 被吊销

Agent 订阅的数据源在付款前被标记为投毒/失效 → 熔断,不为坏数据付费。

算力/模型跑路

供应商合约不在白名单或临时换了收款地址 → 拒绝,钱不出门。

正常工具租约

来源持续可信 → 结算并交付,租约可执行、可审计。

凡是"Agent 自动为外部来源付费"的场景,都需要一个付款前的最后一道闸。

关键代码

强制力写在合约里

① 闸门:来源不 Active 就 revert,支付无法落地

// ProcurementGate (示意)
function settleSpend(bytes32 src, ...) {
  require(
    sourceState[src] == Active,
    "source not active"   // 跳闸 → 0 转移
  );
  /* …授权 + 支付 + 交付… */
}

② 浏览器 keyless 复核:eth_call 直读链上状态

// chain.mjs (示意)
const data = "0x447c24c0" + pad(srcHash); // sourceState(bytes32)
const r = await rpc("eth_call", [{
  to: registry, data
}, "latest"]);
// 0=Unknown 1=Active 2=Challenged 3=Revoked

还内置一次性传输中断注入(?fail=1):动作失败 → 资金停在原地、可重试,证明 fail-closed 不是说说。
