问题

AI Agent 正在自己花钱

它能自己下单、签名、付款。可一旦它被骗、或它买的东西在付款瞬间已经变质,谁来拦?

被注入

提示注入让 Agent 把钱付给错误的合约地址,它自己不会怀疑。

来源失效

报价时来源是安全的,付款前却被吊销/挑战,钱照样打出去。

付款拿不到货

钱已上链,交付却没发生,链上不可逆。

80% 的组织报告过 AI agent 的风险行为(McKinsey)。问题是:交易一旦上链就没法撤销,发现得再快,钱已经没了。来源:McKinsey · OWASP Agentic Top 10

真实会发生的事

你的 Agent买了一份过期数据

你让 AI Agent 去买一份市场数据 API 的调用额度。

  • Agent 调查了供应商,数据源状态正常,报价合理。它决定买。
  • 但在 Agent 做完调查到真正付款的这 30 分钟里,供应商的 API 端点被攻击者替换成了投毒数据。监控者已经把这个数据源标记为不可信。
  • Agent 不知道。它还在用 30 分钟前的"调查结果"。钱照样打出去了,买到的是一堆假数据,而且链上交易没法撤。

这不是假设。88% 运行 AI agent 的组织已经报告过类似安全事件。beam.ai 2026

攻击是怎么发生的

一次典型的来源投毒攻击

现有方案的盲区

决策时 ≠ 付款时

  • Cobo CAW / Pact 白名单:约束的是"Agent 能不能发起这笔操作"。这是决策时的安全。
  • LLM 风控 / 搜索判断:Agent 用模型去查"这个供应商靠不靠谱"。但 LLM 的搜索能力有限,结果可以被投毒。也是决策时的事。
  • 问题是:在 web3 上,交易一旦提交就不可逆。从 Agent 做决定到钱真正打出去,中间可能隔了几分钟,来源状态已经变了。
  • PactFuse 管的是付款时:在交易真正执行的那一刻,在链上重新检查。不依赖 LLM 的判断,用合约里的确定性逻辑。

大多数项目在研究怎么让 LLM 判断得更准,或者加链下限制。但 LLM 本身就有搜索盲区,而且能被投毒。PactFuse 换了一条路:不改进判断,直接在链上执行层加一道硬卡。

我们在哪

PactFuse 的定位

  • 左上角:付款时介入 + 链上判断。只有 PactFuse 在这个位置。
  • LLM 风控 / 打分:决策时判断,链下执行。判断能力受限于 LLM,可被投毒。
  • 审计工具:决策时检查,但拦不住交易本身。
  • 多签/MPC:链上能拦,但只管"谁签了",不管"来源还安不安全"。
解法

付款前,在链上熔断

别的方案管决策时,PactFuse 管付款时。在交易真正落地的那一刻,链上合约重新读来源状态:不安全就直接拦住,一分钱都不会打出去。

熔断

来源被挑战 → 闸门跳开,所有钉住它的支付在付款前被切断。

结算

来源干净 → 在策略边界内授权、支付、交付 artifact。

拒绝

目标不在 Pact 白名单 → 任何东西上链前直接拒绝。

设计原则:有疑必拦。合约查不到"来源安全"这个状态,交易就直接失败。宁可多拦一次,也不放过一笔有问题的付款。

怎么做 · 架构

一次支付的三道关卡

付款前的决策流

每一笔支付,先过这道判断

关键就在 读 sourceState 这一步:合约里写死的判断,不是 AI 猜的。查到不安全,交易直接失败,钱根本打不出去。

现实场景

把三场景映射到真实采购

数据/API 被吊销

Agent 订阅的数据源在付款前被标记为投毒/失效 → 熔断,不为坏数据付费。

算力/模型跑路

供应商合约不在白名单或临时换了收款地址 → 拒绝,钱不出门。

正常工具租约

来源持续可信 → 结算并交付,租约可执行、可审计。

只要是 Agent 自己掏钱买外部服务,就需要一个付款前的最后检查。

"能不能付"写在合约里

// ProcurementGate (示意)

  require(
    sourceState[src] == Active,
    "source not active"   // 跳闸 → 0 转移
  );
  /* …授权 + 支付 + 交付… */
}
